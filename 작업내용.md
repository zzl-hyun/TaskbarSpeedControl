# 작업내용

## 문제

- `Apply` 버튼을 눌러도 작업표시줄 자동 숨김 애니메이션 속도가 실제로 바뀌지 않았음.
- 초기 구현은 `AnimateWindow` IAT 패치 방식이었지만, 현재 Windows 11 작업표시줄 자동 숨김 애니메이션은 이 경로를 거의 타지 않음.
- 그래서 설정 저장은 되지만 실제 Explorer 작업표시줄 애니메이션에는 영향이 없었음.

## 확인한 내용

- 앱 구조:
  - C# WinForms UI: `src/TaskbarAutoHideSpeed`
  - Native hook DLL: `src/TaskbarAutoHideHook`
- Windhawk의 `taskbar-auto-hide-speed` 구현을 확인함.
  - Windhawk는 단순 `AnimateWindow`가 아니라 `TrayUI::SlideWindow`, `GetTickCount`, `Sleep`, Windows 11의 `Taskbar.View.dll` Composition 애니메이션 경로를 사용함.
- 사용자 로그에서 다음이 확인됨:
  - DLL이 Explorer에 주입됨: `InjectSelfIntoExplorer: success`
  - IAT 패치 성공: `HookWorkerThread: patch success`
  - 작업표시줄 모듈 로드 확인: `Taskbar.View.dll=...`
  - 실제 작업표시줄 이동 감지 성공: `Taskbar slide detected: show/hide ...`

## C# 쪽 변경

### `NativeBridge.cs`

- 네이티브 DLL 초기화와 설정 reload 결과를 실제로 확인하도록 변경.
- 기존에는 `_initialize?.Invoke()` / `_reloadSettings?.Invoke()` 결과를 무시해서 실패해도 UI에는 성공처럼 보였음.
- `NativeApplyResult` record 추가.
  - 성공/실패 여부와 메시지를 UI까지 전달함.
- Win32 error 확인을 위해 delegate에 `SetLastError = true` 추가.

### `TaskbarSpeedController.cs`

- `Apply()`와 `RestoreDefaults()`가 `void` 대신 `NativeApplyResult`를 반환하도록 변경.
- 설정 저장, 시작 프로그램 등록, 네이티브 hook 적용 결과를 호출자에게 전달함.

### `MainForm.cs`

- `Apply` 버튼 클릭 후 실제 네이티브 hook 적용 결과를 상태 라벨에 표시하도록 변경.
- 실패 시 실패 메시지가 보이게 함.
- 성공했지만 속도 변화가 없을 경우 fallback hook 경로가 맞지 않을 수 있다는 안내를 표시함.

## Native DLL 쪽 변경

### 기본값/설정 로드 안정화

- `NativeSettings` 기본값 추가:
  - `ShowSpeedup = 250`
  - `HideSpeedup = 250`
  - `FrameRate = 90`
  - `RunAtStartup = TRUE`
- 설정 파일이 없거나 읽기 실패할 때 `show=0 hide=0 fps=0`으로 남지 않도록 방어.
- 설정값 clamp 추가:
  - speedup 최소 1
  - frame rate 1~240
- 설정 로그는 값이 바뀔 때만 찍도록 정리.

### Explorer 주입/오류 로그 개선

- 주입 실패 지점마다 `GetLastError()`를 로그에 남기도록 `LogLastError()` 추가.
- `CreateRemoteThread` 후 remote `LoadLibraryW` 결과를 확인하도록 변경.
- 성공 시 `InjectSelfIntoExplorer: success` 로그 출력.

### IAT hook 확장

- 기존 `AnimateWindow` 외에 다음 API도 IAT 패치 대상으로 추가:
  - `SetWindowPos`
  - `MoveWindow`
  - `GetTickCount`
  - `Sleep`
- 작업표시줄 창 class 감지:
  - `Shell_TrayWnd`
  - `Shell_SecondaryTrayWnd`
  - `Shell_SecondaryTrayWnd2`
  - `SecondaryTrayWnd`
- 작업표시줄 visible area 변화를 계산해서 show/hide 슬라이드 감지.
- 감지 시 로그:
  - `Taskbar slide detected: show ...`
  - `Taskbar slide detected: hide ...`

### 속도 적용 로직

- 작업표시줄 slide가 감지된 thread에서만 `GetTickCount`와 `Sleep`을 조정.
- `GetTickCount`:
  - slide 시작 후 흐른 시간을 `ShowSpeedup` 또는 `HideSpeedup` 비율만큼 빠르게 보이게 함.
- `Sleep`:
  - slide thread에서 sleep 시간을 speedup 비율만큼 줄임.
  - `FrameRate` 기준으로 frame pacing을 조정.
- 최근 추가 수정:
  - 한 show/hide 애니메이션 중 프레임마다 slide 시작점이 다시 잡히지 않게 방어.
  - 같은 thread, 같은 방향의 연속 이동은 첫 감지만 slide 시작으로 사용.

### 모듈 로그 추가

- Explorer 내부에서 taskbar 관련 모듈 로드 상태를 로그로 남김:
  - `taskbar.dll`
  - `Taskbar.View.dll`
  - `ExplorerExtensions.dll`

## 빌드/검증

- 다음 빌드들은 MSBuild로 성공 확인됨:
  - C# Release x64
  - C++ Native DLL Release x64
- 사용 명령:

```powershell
MSBuild.exe src\TaskbarAutoHideSpeed\TaskbarAutoHideSpeed.csproj /p:Configuration=Release /p:Platform=x64 /p:Restore=false /verbosity:minimal
MSBuild.exe src\TaskbarAutoHideHook\TaskbarAutoHideHook.vcxproj /p:Configuration=Release /p:Platform=x64 /verbosity:minimal
```

- 실행/배포는 기존 스크립트 사용:

```powershell
.\build.ps1 -Configuration Release -Platform x64
```

- 이 스크립트는 Explorer를 재시작해서 기존에 잠긴 `TaskbarAutoHideHook.dll`을 교체함.

## 현재 상태

- Explorer 주입 성공.
- IAT 패치 성공.
- 실제 작업표시줄 자동 숨김 show/hide 이동 감지 성공.
- 로그에서 `Taskbar slide detected`가 확인됨.
- 마지막 수정본은 빌드 성공했지만, 당시 실행 중인 Explorer/앱이 DLL을 잡고 있어서 출력 폴더 복사는 실패함.
- 따라서 최종 반영은 아래 명령으로 해야 함:

```powershell
.\build.ps1 -Configuration Release -Platform x64
```

## 확인할 로그

```powershell
Get-Content "$env:APPDATA\TaskbarAutoHideSpeed\hook.log" -Tail 80
```

확인 포인트:

- `InjectSelfIntoExplorer: success`
- `HookWorkerThread: patch success`
- `Taskbar modules: ...`
- `Taskbar slide detected: show ...`
- `Taskbar slide detected: hide ...`

최근 수정 후에는 `Taskbar slide detected`가 애니메이션 프레임마다 너무 많이 찍히지 않고, show/hide 한 번당 더 적게 찍히는 것이 정상.

## 남은 과제

- 현재 구현은 Win32 slide/timing 경로를 잡는 방식.
- 로그상 `Taskbar.View.dll`도 로드되어 있으므로 Windows 11 Composition 애니메이션 경로가 일부 섞여 있을 수 있음.
- 속도 체감이 충분하지 않다면 다음 단계는 Windhawk처럼 PDB 심볼 기반으로 비공개 함수 주소를 찾아 직접 hook하는 방식이 필요함.
  - 예: `TrayUI::SlideWindow`
  - 예: `SharedAnimations::TaskbarCollapseAnimation`
  - 예: `SharedAnimations::TaskbarExpandAnimation`
- 해당 단계는 Microsoft symbol server / DbgHelp / inline hook 구현이 필요해서 안정성 검증이 더 중요함.
